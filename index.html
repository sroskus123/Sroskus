Dynamic content for Fabric servers.

Sroskus Custom Items lets server owners add items, weapons, tools, armor, blocks, recipes, models, textures, and random block drops from JSON files without shipping a new mod build every content update.

Install

Install the mod once on the dedicated server and on every client. Server owners then manage gameplay content from the world folder. Players do not need to manually copy every server texture or JSON file if the server sends the asset cache during join.

Deployment rule: the mod must exist on both sides. The server is authoritative for stats, recipes, drops, block behavior, and armor behavior. The client receives render assets and registry metadata through the sync cache.
Server:
mods/sroskus-item-<version>.jar
world/Sroskus_Customitems/

Client:
mods/sroskus-item-<version>.jar
.minecraft/Sroskus_Customitems/cache/<server-content-hash>/

Config

The config file lives in the normal Minecraft config folder. In dev it appears under run/config/.

# .minecraft/config/sroskus_item.properties
generateexamples=true

When generateexamples is true, the mod creates example JSON files, textures, model templates, armor UV templates, and readmes in each content folder. Keep it enabled for first setup, then disable it if you want a completely hand-managed production folder.

Client Sync

On multiplayer join, the server sends a content manifest during the Fabric configuration phase. The client checks its local cache by hash. If the hash is missing, the client requests the content archive, receives it in chunks, extracts it into the cache, and mounts the generated resource pack.

Different servers are safe. Each server content state is cached under its own hash, so one client can join multiple servers without replacing another server's assets.
.minecraft/Sroskus_Customitems/cache/
  9b47d9.../
    registry/
    textures/
    models/
  f13ac2.../
    registry/
    textures/
    models/

The generated resource pack is created locally from the active cache. This is what makes item icons, block models, item models, and armor layer textures render on the client.

Registry JSON

Each registry JSON file creates one content definition. File names can differ from IDs, but the id must be lowercase and use only letters, numbers, underscores, hyphens, dots, or slashes.

FieldUsed byNotes
idAllStable content ID used by recipes and drops.
typeAllitem, sword, pickaxe, axe, shovel, hoe, helmet, chestplate, leggings, boots, block.
displayNameAllName shown in game.
textureAllRelative PNG path inside textures/.
modelItems, tools, armor items, blocksOptional JSON path inside models/. If missing, a default model is generated.
stackSizeItems, blocksClamped from 1 to 64.
durabilityTools, weapons, armorRuntime behavior reads this value from the definition.
attackDamage, attackSpeed, miningSpeedWeapons and toolsControls combat and mining behavior.
armor, toughness, knockbackResistanceArmorControls equipped armor attributes.
armorTextureArmorShared armor layer filename prefix for a full armor set.
hardness, resistance, lightLevelBlocksBlock strength and emitted light. Light is clamped from 0 to 15.

Recipes

Recipes are defined inside registry JSON. Custom IDs can reference other custom content from the same content folder, or vanilla and modded namespaced IDs such as minecraft:diamond.

Shaped

{
  "type": "shaped",
  "pattern": ["AAA", " B ", " B "],
  "keys": {
    "A": "ruby_ingot",
    "B": "minecraft:stick"
  },
  "count": 1
}

Shapeless

{
  "type": "shapeless",
  "ingredients": [
    "ruby_block"
  ],
  "count": 9
}

Drops

Drop rules apply after a successful survival block break. Creative-mode block breaks do not trigger custom drops.

{
  "block": "minecraft:cobblestone",
  "drop": "ruby_sword",
  "amount": 1,
  "chance": 5.0
}
  • block must be a valid namespaced block ID.
  • drop can be a custom content ID or a vanilla/modded item ID.
  • chance is a percent value from 0 to 100.
  • Invalid drop files are logged and skipped instead of crashing the server.

Textures & Models

Textures and models are copied into the generated resource pack on the client. Keep asset paths lowercase and relative. Do not use drive letters, absolute paths, or ...

Item, Tool, Weapon, and Block Icons

  • Use square PNG files such as 16x16, 32x32, 64x64, or larger square sizes.
  • If model is missing, the mod generates a flat item or block model from texture.
  • If model is present, the JSON is copied from models/ and can use vanilla item model fields, including parent, textures, display, elements, and UV arrays.
{
  "parent": "minecraft:item/handheld",
  "textures": {
    "layer0": "sroskus_item:item/weapons_and_tools/ruby_sword"
  }
}

Worn Armor UV

Armor uses vanilla biped armor layer textures, not custom animated geometry. A full armor set should share one armorTexture value, then provide both layer files.

Sroskus_Customitems/textures/models/armor/ruby_layer_1.png
Sroskus_Customitems/textures/models/armor/ruby_layer_2.png
  • layer_1 is used by helmet, chestplate, and boots.
  • layer_2 is used by leggings.
  • Supported layer sizes are vanilla-compatible 2:1 PNGs: 64x32, 128x64, 256x128, or larger same-ratio images.
  • If layer files are missing, the mod generates fallback layers so the server keeps running.

Reload Flow

Use the reload command after editing JSON, textures, models, or drops. Both command aliases are available.

/sroskus_c reload
/sroskus_ce reload
  1. Edit files in the server world Sroskus_Customitems folder.
  2. Run /sroskus_c reload.
  3. The server validates JSON and updates recipes, drops, stats, blocks, and slots.
  4. Online clients receive the new manifest and cache update.
  5. The generated datapack and generated resource pack are rebuilt.

New live content uses pre-registered hot slots such as sroskus_item_hot_sword_001. This is what allows reloads without restarting Minecraft. Restarting keeps assignment stable when ordering stays stable.

Troubleshooting

Item shows as missing texture

Check that texture points to an existing PNG under textures/, uses lowercase characters, and does not include a namespace. Example: weapons_and_tools/ruby_sword.png.

Custom model does not apply

Check that model points to a JSON under models/. The model's texture references should use the generated namespace path such as sroskus_item:item/weapons_and_tools/ruby_sword.

Armor item icon is correct, but worn armor is wrong

Make sure every armor piece in the set has one armorTexture value and that both armor layer files exist under textures/models/armor/.

Player cache looks stale

Delete the specific cache hash folder, or delete .minecraft/Sroskus_Customitems/cache/. The next server join downloads the current server content again.

JSON errors

Invalid files are logged with the relative path. Fix the file, then run /sroskus_ce reload. The server should skip bad files instead of crashing.
